What Is a Project? — 4 Defining Characteristics (TUCU)

A project is a temporary, unique, cross-functional endeavour that involves uncertainty and delivers change through outputs, outcomes, and benefits.

Temporary — has a defined start and end; the project team is disbanded when the project closes. The outputs it creates may last permanently, but the project itself does not.
Unique — not routine or repetitive. Each project is a one-off piece of work. Even similar projects differ in context, people, or constraints.
Cross-functional — involves people from different departments, specialisms, or organisations working together (e.g. IT, Finance, Marketing, external suppliers).
Uncertain — projects introduce change, and change brings risk and unknowns that must be actively managed.

Project vs Business As Usual (BAU)
Project Temporary · Unique · Change-driven · Cross-functional · Uncertain
BAU Ongoing · Repetitive · Stable · Department-based · Predictable
Example: Running a helpdesk = BAU. Upgrading the helpdesk system = Project.
🧠 MNEMONIC — 4 Project Characteristics:
Abbreviation: TUCU → "The Unicorn Crossed Uncharted terrain"
Temporary · Unique · Cross-functional · Uncertain
⚡ Key trap: "Temporary" refers to the PROJECT — not the outputs. A bridge built by a project lasts decades; the project that built it ended at handover.
⚡ Key trap: "Cross-functional" = drawing people from multiple teams/departments — NOT that one person can do everything.

4 Integrated Elements of PRINCE2 7

PRINCE2 7 is built on four integrated elements that must work together to deliver successful projects:

1. Principles — 7 mandatory behaviours that define what it means to be using PRINCE2. Cannot be tailored away.
2. Practices — 7 management areas (formerly "Themes") that address how specific aspects of the project are managed (Business Case, Plans, Quality, etc.).
3. Processes — 7 process groups defining what happens at each stage of the project lifecycle (SU → DP → IP → CS/MP → SB → CP).
4. People — the new element in PRINCE2 7; recognises that leadership, team dynamics, stakeholder engagement, and culture are central to success.
🧠 MNEMONIC — 4 Integrated Elements:
Sentence: "Practical People Plan Projects"
Principles · People · Practices · Processes
⚡ Key: PRINCE2 7 added "People" as a formal element — it did NOT exist as a named element in PRINCE2 6.
⚡ Only Principles are non-negotiable. Practices, Processes, and People-related activities can all be tailored.

Delivery Approaches: Linear, Iterative & Hybrid

PRINCE2 7 is delivery-approach agnostic — it can work with all three approaches.

Linear (Predictive / Sequential / Waterfall)
Each phase must be fully completed and approved before the next phase begins. Requirements are defined upfront. Best when: requirements are clear, stable, and well-understood; regulatory sign-off is required at each stage.
Example: Building a bridge — foundation must be done before walls, walls before roof.

Iterative / Incremental (Adaptive / Agile)
Work is organised into timeboxed iterations (sprints). Each iteration delivers a potentially usable increment and generates feedback that refines future iterations. Best when: requirements are unclear or evolving; fast learning and adaptation are needed.
Example: Building a mobile app — release MVP, gather user feedback, improve in the next sprint.

Hybrid
Combines predictive and adaptive elements within a single project. The nature of the work determines which parts use which approach. Integration points between predictive and adaptive streams must be carefully managed to avoid clashes in pace and governance.
Example: Fixed regulatory reporting (linear) combined with a flexible UX development track (iterative).

Factors that determine the right approach:
• Requirement clarity — clear & stable → linear; unclear & evolving → iterative
• Organisational experience — limited agile experience → avoid pure iterative
• Regulatory constraints — mandatory phase-gate sign-offs → linear elements required
• Uncertainty level — high uncertainty → iterative; low uncertainty → linear
• Nature of deliverables — physical/construction → linear; software/digital → often iterative
🧠 MNEMONIC — 3 Delivery Approaches:
"Lions Ignore Heat" → Linear · Iterative · Hybrid
Linear = "WATERFALL" — Water only flows downhill, never back up. Phases go in one direction.
Iterative = "SPIRAL" — Loops get better each time. Feedback drives the next cycle.
Hybrid = "MOSAIC" — Different pieces, different styles, assembled into one picture.
⚡ Key: In hybrid, INTEGRATION POINTS are where the two streams synchronise — these are critical governance checkpoints.
⚡ Key: Governance structures do NOT remain identical in hybrid vs pure linear — they must accommodate both cadences.

Project Management Definition & Project Performance Aspects

Project Management Definition (PRINCE2):
The planning, delegating, monitoring and control of all aspects of a project, and the motivation of those involved, to achieve the project objectives within the expected performance targets for:
Time Cost Quality Scope Risk Benefits Sustainability

These 6+1 are the Performance Aspects — what the PM monitors and controls. They overlap with Tolerances (the permitted deviations from targets for each aspect).

Key distinction — the 4-part definition:
• Planning — deciding what to do, who does it, when, and how
• Delegating — assigning work to the right people at the right level
• Monitoring — checking that work is on track against the plan
• Controlling — taking corrective action when deviations are forecast
🧠 MNEMONIC — PM Definition:
"Please Don't Miss Control" → Planning · Delegating · Monitoring · Controlling
⚡ Key exam trap: Options that say "planning, delegating, monitoring, control of repetitive activities" describe BAU — NOT project management. PM targets TIME, COST, QUALITY, SCOPE, BENEFITS, RISK, SUSTAINABILITY.
⚡ Key: Sustainability (environmental, social, economic impact) is the NEW 7th performance aspect added in PRINCE2 7. It sits alongside the other 6, not above them.

7 PRINCE2 Principles

1. Continued business justification
The project must have a justifiable reason to start and continue throughout its duration. The justification is documented in the Business Case and verified at every stage boundary.
Layman terms: Always ask 'Is this still worth doing?' If building a house suddenly costs 5x more than it's worth, stop building. Don't finish it just because you started.

2. Learn from experience
Lessons are sought, recorded, and acted upon throughout the project and after its completion. The Lessons Log is maintained at all times.
Layman terms: Don't repeat past mistakes. If you burned the cake last time because the oven was too hot, write that down so whoever bakes next knows to lower the temperature.

3. Defined roles and responsibilities
The project has clearly defined and agreed roles with areas of responsibility that consider the interests of the business, users, and suppliers.
Layman terms: Everyone needs to know exactly what their job is. If you're organizing a wedding, you must know who is bringing the cake, who is paying, and who is choosing the music.

4. Manage by stages
The project is planned, monitored, and controlled stage by stage. Detailed planning for the short term, with high-level planning for the future.
Layman terms: Break the big scary project into bite-sized chunks (stages). Don't plan every detail of a 2-year road trip today; just plan how to get to the first city, then stop, check your budget, and plan the next leg.

5. Manage by exception
Each level of project management has authority within established tolerance limits. Exceptions are escalated to a higher level only when tolerances are forecast to be exceeded.
Layman terms: Let people do their jobs without micromanaging. Only bother the boss if things go seriously wrong (like running out of money or being a month late). Otherwise, you handle it.

6. Focus on products
PRINCE2 focuses on defining and delivering products – particularly on the quality of requirements. The project delivers products, it does not just carry out activities.
Layman terms: Care about the final result, not just being busy. Instead of saying 'We will paint for 5 hours', say 'We will deliver a perfectly painted blue wall'.

7. Tailor to suit the project
PRINCE2 is tailored to the specifics, scale, complexity, importance, capability, and risk of the project. There is no one-size-fits-all.
Layman terms: Use common sense to adjust the rules. You wouldn't use the same massive paperwork to organize a backyard BBQ as you would to build a skyscraper.
🧠 MNEMONIC — 7 Principles:
Sentence: "Just Learn Roles, Stages Except Products Tailor"
Just → Continued business justification  ·  Learn → Learn from experience  ·  Roles → Defined roles and responsibilities
Stages → Manage by stages  ·  Except → Manage by exception  ·  Products → Focus on products  ·  Tailor → Tailor to suit
⚡ Key: Principles are the ONLY element that CANNOT be tailored — they apply to every project.

The "People" Element (PRINCE2 7th Edition)

People are central to the success of any project. PRINCE2 7 places them at the heart of the method. Key focus areas include:
Leadership and Management: Balancing the need to lead (inspiring, motivating) with management (planning, organizing).
Team Dynamics: Understanding how teams form, develop, and work together effectively.
Stakeholder Engagement: Communicating and building relationships with anyone affected by the project.
Organizational Culture: Recognizing that the project operates within the wider culture of the organization and adapting accordingly.
Change Readiness: Preparing people for the change the project will deliver, managing resistance, and using emotional intelligence.

7 PRINCE2 Practices (Themes)

Business Case Why? Justification and benefits of the project. (Layman: The main reason we're spending money. E.g. 'We spend $500 on a mower to save $1000 a year on landscaping'.)
Organization Who? Roles and responsibilities in the project. (Layman: Knowing exactly who is the boss, who does the work, and who checks if the work is good.)
Quality What? Defining and controlling product quality. (Layman: Making sure the thing actually works. If a customer ordered a waterproof phone, plan to test it in water before handing it over.)
Plans How? When? How much? Planning management levels. (Layman: The roadmap. Knowing who does what, when it needs to be finished, and how much money it will consume.)
Risk What if? Risk identification and management. (Layman: Planning for surprises. 'What if it rains on our outdoor concert?' -> We buy a tent just in case.)
Change What is the impact? Managing changes and issues. (Layman: Handling new ideas safely. If the client suddenly wants a pool instead of a garage, calculate the extra cost before saying 'yes'.)
Progress Where are we? Monitoring and controlling progress. (Layman: Checking the map to see if we are lost. Tracking if we are on schedule and within budget.)
🧠 MNEMONIC — 7 Themes / Practices:
Abbreviation: BOQPRCP → "Big Orange Quail Pecks Ripe Cherries Patiently"
Each practice answers ONE question:
Business Case → Why?  ·  Organization → Who?  ·  Quality → What?
Plans → How? When? How much?  ·  Risk → What if?
Change → What's the impact?  ·  Progress → Where are we?
⚡ Key: In PRINCE2 7, the 7 Themes were renamed to Practices — same content, new name.

7 PRINCE2 Processes

SU – Starting Up a Project
Before initiation. Appointing the executive and project manager, developing the project brief, selecting the approach.
Layman terms: The 'Idea' phase. Just doing a quick check if the idea is even realistic before spending real money on planning.

DP – Directing a Project
Project Board only. Authorizing initiation, stages, closure. Handling exceptions.
Layman terms: The Bosses' job. They don't do the daily work, they just sit at the top giving the green light (or red light) at major checkpoints.

IP – Initiating a Project
Creating the PID (Project Initiation Documentation). Establishing project controls.
Layman terms: The deep planning phase. This is where you sit down and write the detailed rulebook, budget, and schedule before the actual building starts.

CS – Controlling a Stage
Day-to-day management by the Project Manager: assigning work packages, monitoring.
Layman terms: The Manager's daily grind. Handing out tasks to the workers, dealing with small daily problems, and keeping an eye on the clock.

MP – Managing Product Delivery
The Team Manager accepts, executes, and delivers work packages.
Layman terms: The actual building phase. The team goes to their desks (or construction site) and builds the actual product as instructed.

SB – Managing a Stage Boundary
Reporting to the board, updating plans, requesting authorization for the next stage.
Layman terms: The 'End of the level' review. The manager prepares a report card of the work just finished and asks the bosses for money to start the next level.

CP – Closing a Project
Handing over products, evaluating benefits, recommending closure.
Layman terms: Packing up. Handing the keys to the customer, throwing a closing party, and writing down what went well and what sucked.
🧠 MNEMONIC — 7 Processes:
Sentence: "Some Individuals Do Create Many Splendid Cakes"
Some → SU Starting Up a Project  ·  Individuals → IP Initiating a Project
Do → DP Directing a Project ⟲ (runs throughout the ENTIRE project)
Create → CS Controlling a Stage (PM day-to-day)  ↕  Many → MP Managing Product Delivery (Team Manager)
Splendid → SB Managing a Stage Boundary  ·  Cakes → CP Closing a Project
⚡ Key: DP ⟲ = Project Board operates across ALL stages. CS ↕ MP = PM and Team Manager work in PARALLEL within each stage.

Project Management Team Structure

Project Board
• Executive – responsible for the Business Case, has the ultimate say
• Senior User – represents the users of the products
• Senior Supplier – represents the suppliers of resources

Project Manager – manages stages day-to-day
Team Manager – manages the work of production teams
Project Assurance – independent monitoring (not the PM!)
Project Support – administrative support
Change Authority – can be delegated by the Project Board
🧠 MNEMONIC — Team Structure:
Project Board = 3 roles: Executive · Senior User · Senior Supplier → "ESS"
Below the Board: PM (Project Manager) → day-to-day management
PM ↔ TM (Team Manager) → executes Work Packages
Independent: PA (Project Assurance) → oversight for the Board (NOT the PM!)
Administrative: PS (Project Support) → assists PM (CAN be combined with PM)
⚡ Key: Project Assurance can NEVER be delegated to the PM — conflict of interest!
⚡ Key: The Executive owns the Business Case — the PM only develops it on their behalf.

Tolerances (7 dimensions)

Time Cost Scope Quality Risk Benefits Sustainability

Tolerances define permissible deviations from the plan. Exceeding a tolerance = escalation to a higher level. PRINCE2 7 introduced Sustainability to focus on environmental, social, and economic impacts.
🧠 MNEMONIC — 7 Tolerances:
Abbreviation: TCSQRBS → "The Cat Sat Quietly Reading Books Silently"
Time · Cost · Scope · Quality · Risk · Benefits · Sustainability
⚡ Key: Exceeding a tolerance = MANDATORY escalation to the next level of management. No exceptions!

Reports: Checkpoint Report, Highlight Report, End Stage Report, End Project Report, Exception Report
Other: Project Brief, PID, Business Case, Product Description, Work Package, Configuration Item Record
🧠 MNEMONIC — Key Management Products:

Registers / Logs: "RQILD" → Risk · Quality · Issue · Lessons · Daily Log
Reports: "CHEEE" → Checkpoint · Highlight · End Stage · End Project · Exception

Risk Management Procedure (5 steps)

1. Identify – identify risks and their context
2. Assess – probability × impact
3. Plan – select risk responses
4. Implement – act according to the plan
5. Communicate – throughout the process
🧠 MNEMONIC — 5 Steps of Risk Management:
Abbreviation: IAPIC → "I Always Plan In Color"
Identify → Assess → Plan → Implement → Communicate
⚡ Communicate = the only step that runs CONTINUOUSLY throughout the process (not just after Implement).
⚡ Risk in PRINCE2 = Threats (negative) OR Opportunities (positive). Both must be managed!
⚡ Risk Owner = manages the risk. Risk Actionee = CARRIES OUT specific actions on behalf of the Risk Owner.

Business Case Practice — Dis-benefits, Ownership & Content

The Business Case is the central justification document for the project. It is created in Starting Up a Project (outline) and fully developed during Initiating a Project, then reviewed at every stage boundary.

Business Case content includes:
• Reasons for the project (why are we doing this?)
• Business options considered (do nothing / do minimum / do something)
• Expected benefits — measurable improvements from using the project's outputs
• Dis-benefits — perceived negative effects that must also be measured and managed
• Timescales, costs, investment appraisal (ROI, NPV, payback period)
• Major risks

Dis-benefits are negative consequences that stakeholders perceive as adverse outcomes. Example: a new IT system saves £500k/year (benefit) but causes 3 months of reduced productivity during transition (dis-benefit). Both must be documented.

Ownership: The Executive owns the Business Case. The PM may develop it on the Executive's behalf, but ultimate accountability always remains with the Executive. This is a classic exam trap.
🧠 MNEMONIC — Business Case:
"Benefits Defeat Costs" → Benefits · Dis-benefits · Costs (all three must be quantified)
⚡ Key: Dis-benefits are NOT risks — they are expected negative outcomes that we plan for, not uncertain events.
⚡ Key: If the Business Case is no longer valid, the ONLY correct PRINCE2 answer is to recommend premature closure.
⚡ Key: The Business Case is "living" — it must be updated and reviewed at EVERY stage boundary, not just at the start.

Plans Practice — WBS, Dependencies & Planning Horizon

Work Breakdown Structure (WBS)
A hierarchical decomposition of the project scope into progressively smaller and more manageable components. It starts with the final deliverable and breaks it down level by level. The WBS supports product-based planning by identifying all products before determining activities.

Dependency Types:
Mandatory (Hard) Legally or physically required — cannot be changed. Example: permit must be granted before excavation can begin (regulatory law).
Discretionary (Soft) Best practice or preferred sequence chosen by the team — could technically be changed. Example: "we prefer to complete design before coding, though we could do both in parallel."
External Dependencies on parties outside the project — e.g., a supplier delivery window.

Planning Horizon
The extent into the future for which reliable detailed planning can be produced. PRINCE2's Manage by Stages principle directly addresses this: only the current stage has a detailed plan; future stages are planned at a high level until sufficient information exists.
🧠 MNEMONIC — Dependency Types:
"Must Do External" → Mandatory · Discretionary · External
⚡ Key exam trap: "Legal/regulatory requirement" = MANDATORY. "Industry best practice" = DISCRETIONARY.
⚡ Planning horizon = "How far ahead can we plan in detail?" → Answer: only the current stage (PRINCE2 solves this via stage gates).

Quality Practice — Product Description Elements & Quality Controls

Product Description — the key quality planning document for each product. Key elements:
Purpose Why the product is needed / intended use
Composition What the product is made of
Derivation Source products / inputs
Format & Presentation How it is presented
Quality Criteria Measurable characteristics and performance standards the product MUST achieve
Quality Tolerance Acceptable deviation from the quality criteria
Quality Method How the product will be checked / tested
Quality Skills Required Who checks it

Three Quality Activities:
Quality Planning — defining standards and criteria upfront (in Product Description)
Quality Control — checking deliverables against specifications to detect defects; done by the team
Quality Assurance — independent checking that the PM's processes are being followed correctly; done by Project Assurance (NOT the PM)
🧠 MNEMONIC — Product Description elements:
"Please Check Deliveries For Quality Tolerances & Methods"
Purpose · Composition · Derivation · Format · Quality criteria · Tolerance · Method
⚡ Key: Quality criteria = what standards must be met. Quality tolerance = how much deviation is OK.
⚡ Quality Control = team checks the PRODUCT. Quality Assurance = independent check of the PROCESS.

Change Practice — Issue Types, Issue Register & Change Management Approach

PRINCE2 classifies all formal project issues into three types:

Request for Change (RFC) A proposal to change a baselined product. Example: client wants to add a new feature. Must go through Change Authority for approval.

Off-Specification A product that does not or will not meet its agreed specification. Example: delivered software fails a test. Must be corrected or formally accepted as a concession.

Problem / Concern Any other matter requiring management attention that is neither an RFC nor an Off-Specification. Example: a key team member resigns.

Issue Register — captures ALL issues (RFC, Off-Spec, Problem/Concern), their status, priority, and resolution actions. Created in IP, maintained by the PM.

Issue Report — a formal document for a specific issue that needs escalation, detailed impact analysis, or a formal decision. Not every issue needs a Report — minor issues stay in the Register.

Change Management Approach — created during IP, defines HOW changes will be identified, assessed, controlled, and recorded. Includes: change budget, role of the Change Authority, escalation thresholds.
🧠 MNEMONIC — 3 Issue Types:
"Rich Old People" → Request for Change · Off-Specification · Problem/Concern
⚡ Key: RFC = someone WANTS something different. Off-Spec = something IS different (from spec). Problem = everything else.
⚡ Key: Issue Register = ALL issues (ongoing log). Issue Report = formal document for a SPECIFIC complex issue.
⚡ Change Authority can be delegated by the Project Board to the PM or a change panel to approve minor RFCs.

Risk Response Types — Threats & Opportunities

PRINCE2 recognizes five response types for threats and four for opportunities:

Responses to THREATS (negative risks):
Avoid Change the plan to remove the threat or protect objectives from its impact. Most powerful response.
Reduce Take action to reduce probability and/or impact. Most common response.
Transfer Shift financial consequences to a third party (e.g. insurance, penalty clauses).
Accept Acknowledge the risk and monitor — do nothing proactively. Used when cost of response exceeds impact.
Share Allocate risk between multiple parties (often in a partnership).

Responses to OPPORTUNITIES (positive risks):
Exploit Take action to make the opportunity definitely happen.
Enhance Increase probability and/or impact of the opportunity.
Share Partner with another to capture the opportunity jointly.
Accept Take advantage if it occurs, but do nothing to pursue it.

Risk Management Approach — the management product that defines how the project will apply the IAPIC procedure, risk appetite, risk tolerance, roles (Risk Owner, Risk Actionee), and tools.
🧠 MNEMONIC — Threat Responses:
"Avoid Really Tough Adversity, Share" → Avoid · Reduce · Transfer · Accept · Share
⚡ Avoid = remove the cause of the threat. Reduce = lower prob/impact. Transfer = financial shift (insurance). Accept = do nothing. Share = partner.
⚡ Key: "Avoid" does NOT mean ignore — it means CHANGE THE PLAN to remove the threat conditions.

Key Process Activities — IP, CS, MP, SB & CP in Depth

IP — Initiating a Project: what the PID contains
The Project Initiation Documentation (PID) is the "contract" for the project. It includes: Business Case · Project Approach · Management approaches (Risk, Quality, Change, Communication) · Project Plan · Project Controls · Role descriptions · Benefits Management Approach.

CS — Controlling a Stage: Stage Status Review
The PM periodically reviews stage status to confirm: (1) are planned products being delivered? (2) is forecasted performance within approved tolerances? If yes → continue. If no → corrective action or Exception Report.

MP — Managing Product Delivery: Work Package flow
1. PM authorizes Work Package (in CS) → 2. Team Manager accepts Work Package (confirms achievable) → 3. Team delivers, checks quality against Product Description → 4. Team Manager sends Checkpoint Reports to PM → 5. Team Manager delivers completed Work Package back to PM.

SB — Managing a Stage Boundary: End Stage Report
At stage end, PM produces: End Stage Report (how did this stage perform?) + Next Stage Plan (what does the next stage look like?). The Project Board reviews both and decides: authorize / request Exception Plan / close project.

CP — Closing a Project: key requirements
Before closure can be completed: customer must accept the project product; operational/support arrangements must be in place; team members must be formally released from project assignments; access to project systems must be revoked. PM produces the End of Project Report.
🧠 MNEMONIC — CP Closure Checklist:
"All Operations Release Resources" → Acceptance by customer · Operational support in place · Released team members · Revoked system access
⚡ Work Package flow: PM authorizes → TM accepts (feasibility check!) → TM delivers → Checkpoint Report → PM receives complete WP.
⚡ End Stage Report = backward look (how did the stage go?). Next Stage Plan = forward look (what's next?).
⚡ End of Project Report ≠ End Stage Report — the End of Project Report is ONLY produced in Closing a Project (CP), not at every stage.

Stakeholders — Definition, Types & Engagement

A stakeholder in PRINCE2 is any individual, group, or organisation that can affect, be affected by, or perceive itself to be affected by the project. This is a deliberately broad definition — stakeholders are NOT limited to end-users or beneficiaries.

Examples of stakeholders:
• Project team members (PM, Team Manager, specialists)
• End-users who will operate the final product
• The sponsoring organisation (Executive, Project Board)
• Suppliers and contractors
• Regulators and government bodies
• Communities or third parties impacted by the project's outputs

Stakeholder Engagement in PRINCE2 7:
Stakeholder engagement is driven by the Communication Management Approach (created during IP). It defines: who the stakeholders are; what information they need; when and how they will receive it; and who is responsible for communication. In PRINCE2 7, stakeholder engagement is also embedded in the People element, recognising that relationships, influence, and trust are central to project success.

Senior User vs Stakeholder: The Senior User on the Project Board represents ALL users as a stakeholder group — but stakeholders extend far beyond the Project Board.
🧠 MNEMONIC — Stakeholder definition:
"Affect · Affected · Perceive" — any of the three is enough to be a stakeholder.
⚡ Key exam trap: "Only project beneficiaries" = WRONG. "All corporate organisation members" = WRONG (too broad). Correct = all individuals involved in OR impacted by the project.
⚡ Key: The Communication Management Approach (not a "plan") defines HOW stakeholders are engaged. Created in IP, owned by the PM.

Management Stages vs Technical Stages

PRINCE2 uses the word "stage" in two distinct senses that are frequently confused in exam questions:

Management Stages — formal control points used by the Project Board to govern the project:
• Sequential — one management stage must be authorised and fully completed before the next begins.
• The Project Board authorises only one management stage at a time.
• At least 2 management stages are required by PRINCE2 (the Initiation Stage + at least one Delivery Stage).
• Correspond to the "Manage by Stages" principle.

Technical Stages (Specialist Stages) — work streams defined by the type of work being done:
• Examples: design, build, test, deploy — or procurement, construction, commissioning.
• Can overlap — e.g., design and procurement can run in parallel.
• NOT governance checkpoints; the Project Board does not formally authorise them.
• Reflect the technical/specialist logic of the work, not the governance structure.

Relationship: Multiple technical stages may exist within a single management stage. The two concepts operate independently — the Project Board controls via management stages; specialists organise work via technical stages.
🧠 MNEMONIC — Management vs Technical Stages:
Management = "Must be Sequential" → Board governance, never overlap.
Technical = "Teams can Overlap" → Specialist work, can run in parallel.
⚡ Key exam answer: Technical stages MAY overlap. Management stages CANNOT overlap.
⚡ Key: A management stage is about WHEN the board reviews; a technical stage is about WHAT type of work is being done.

Tolerance Hierarchy — 3 Levels of Management & Escalation

PRINCE2 applies the Manage by Exception principle through a three-tier delegation and escalation chain. Each level sets tolerances for the level below; if those tolerances are forecast to be exceeded, the lower level must immediately escalate — it cannot authorise the overshoot itself.

Level 1 — Corporate / Programme Management
  ↓ Sets project-level tolerances (in the Project Mandate / Project Brief)
Level 2 — Project Board
  ↓ Sets stage-level tolerances (in the Stage Plan authorisation)
Level 3 — Project Manager
  ↓ Sets work package tolerances (in Work Packages issued to Team Managers)
Level 4 — Team Manager

Escalation mechanism:
• Team Manager forecasts WP tolerance breach → reports to PM (via Checkpoint Report or informally)
• PM forecasts stage tolerance breach → sends Exception Report to Project Board
• Project Board forecasts project-level tolerance breach → escalates to Corporate / Programme Management

Exception Plan: If the Project Board decides to continue after receiving an Exception Report, it may request an Exception Plan — a revised plan replacing the current Stage Plan. The Exception Plan must be authorised before work continues.
🧠 MNEMONIC — 3-Tier Escalation:
"Corporate Boss Punishes Teams" → Corporate → Board → PM → Team Manager
⚡ Key Q6010 trap: If PROJECT-level tolerances are forecast to be exceeded, the Project Board escalates to Corporate / Programme Management — NOT to the Executive alone.
⚡ Key Q6008: "Setting tolerances at ALL THREE plan levels" = direct application of Manage by Exception principle.
⚡ Exceeding stage tolerance → Exception Report → Project Board. NOT an End Stage Report — that comes at normal stage end.

Quality Review Meeting — Purpose, Roles & Outcomes

A quality review (or quality review meeting) is a structured technique used to inspect a product against its Product Description quality criteria. It is the primary quality control activity for document-type products.

Roles in a quality review:
Chair Manages the meeting process; ensures it stays focused and productive.
Producer Created the product being reviewed; presents and responds to comments.
Reviewers Check the product against its quality criteria; prepare a list of questions and defects before the meeting.
Administrator Records outcomes, actions, and agreed corrections.

3-phase structure:
1. Preparation — reviewers receive the product and Product Description; each prepares a list of questions/defects independently.
2. Review Meeting — reviewers raise defects; the group discusses each; agreed actions are recorded for each defect (who fixes it, by when).
3. Follow-up — producer corrects defects; reviewer(s) confirm corrections are satisfactory; product is signed off in the Quality Register.

Key outcome: The review results in agreed actions per defect — NOT an immediate fix, NOT an automatic escalation to an issue. Only unresolved or disputed defects may be raised as formal issues.
🧠 MNEMONIC — Quality Review Outcome:
"Agree the Actions, then Fix" → Preparation · Review (Agree Actions) · Follow-up (Fix & Sign off)
⚡ Key Q6014: Outcome of the review meeting = agreed actions recorded per defect. NOT "a fix is devised" (fixes come in follow-up). NOT "an issue is logged" (issues only if unresolved).
⚡ Quality Register records all planned and completed quality activities — including review results and sign-offs.

Progress Controls — Time-Driven vs Event-Driven

PRINCE2 classifies management controls into two categories based on what triggers them:

Time-Driven Controls — produced at regular, agreed intervals regardless of whether anything specific has happened:
Checkpoint Report Produced by the Team Manager at agreed intervals → sent to the Project Manager. Reports work package progress.
Highlight Report Produced by the Project Manager at agreed intervals → sent to the Project Board. Reports stage progress, forecasts, and issues.

Event-Driven Controls — produced when a specific event occurs:
Exception Report Triggered when a tolerance is forecast to be exceeded → PM to Project Board.
End Stage Report Triggered at the end of each management stage → PM to Project Board.
End Project Report Triggered at project closure → PM to Project Board.
Lessons Report Triggered at stage end or project end (when relevant) → for distribution to corporate/programme.

Summary — report flow:
Team Manager → PM: Checkpoint Report (time-driven, frequent)
PM → Project Board: Highlight Report (time-driven, regular) | Exception Report (event-driven, crisis)
🧠 MNEMONIC — Time-Driven = 2 reports (CH):
"Clocks Hit regularly" → Checkpoint Report · Highlight Report
Everything else (End Stage, End Project, Exception, Lessons) = Event-Driven.
⚡ Key Q6017 trap: "Highlight Reports and End Stage Reports" = WRONG (End Stage is event-driven). Correct = Checkpoint + Highlight.
⚡ Highlight Report frequency is agreed between PM and Project Board in the Communication Management Approach — it is NOT necessarily weekly.

🧠 Mnemonics — How to remember 7+7+7

7 PRINCIPLES:
"Just Learn Roles, Stages Except Products Tailor"
Just → Continued business justification  ·  Learn → Learn from experience
Roles → Defined roles and responsibilities  ·  Stages → Manage by stages
Except → Manage by exception  ·  Products → Focus on products
Tailor → Tailor to suit the project
7 THEMES / PRACTICES — each answers one question:
Business Case → Why?
Organization → Who?
Quality → What? (criteria)
Plans → How? When? How much?
Risk → What if?
Change → What's the impact?
Progress → Where are we?
English abbreviation: BOQPRCP → "Big Orange Quail Pecks Ripe Cherries Patiently"
7 PROCESSES — story of a project from start to finish:
SU → IP → DP ⟲ → CS ↕ MP → SB → CP
DP ⟲ = Project Board operates throughout the project (overlaps with other processes).
CS ↕ MP = PM (CS) and Team Manager (MP) work in parallel during each stage.
Mnemonic: "Some Individuals Do Create Many Splendid Cakes"

⚠️ Top 10 Exam Traps

1. The pass mark is 55% (33/60) — not 60%!
Many older materials state 60% — that's either an error or relates to Practitioner. Foundation = 55%.
2. Project Assurance CANNOT be delegated to the PM.
Project Assurance is the responsibility of the Project Board. Delegating to the PM = conflict of interest. It can be delegated to others, but NEVER to the Project Manager.
3. "Accountable" ≠ "Responsible" — read the last sentence of the question!
"Who is accountable?" → Executive / Board. "Who is responsible?" → PM or Team Manager. Always focus on the last sentence of the question.
4. The Executive owns the Business Case, not the PM.
The PM develops the Business Case on behalf of the Executive, but the Executive is its owner and is ultimately accountable for it.
5. SU does NOT create the PID — IP does!
SU outputs: Project Brief + outline Business Case + appointing the PM. The full PID (Project Initiation Documentation) is only created during IP (Initiating a Project).
6. Exception Report ≠ Exception Plan — two separate steps.
Exception Report: PM informs the Board of a problem. Exception Plan: PM creates a new plan only upon the Board's request. Don't confuse the order.
7. A project without justification = close the project.
PRINCE2 always protects the Business Case. If a project is no longer viable, the correct answer is a recommendation to close it — not workarounds.
8. Checkpoint Report goes to the PM, not the Project Board.
Checkpoint Report: Team Manager → PM (work package progress).
Highlight Report: PM → Project Board (stage progress).
9. Principles are mandatory and non-negotiable.
Tailoring applies to the processes, themes, and management products — it never exempts you from the 7 principles.
10. Choose the most "PRINCE2" answer, not just "common sense".
The exam tests PRINCE2 methodology, not general managerial common sense. An answer might be logical "in real life" but wrong in PRINCE2.

🔀 Frequently Confused Terms

Output → Outcome → Benefit
Output = tangible product delivered by the project (e.g., a new IT system)
Outcome = change resulting from the use of the product (e.g., faster customer service)
Benefit = measurable business value (e.g., saving $500k annually)
💡 Trick: Cause and effect order: Output → Outcome → Benefit.
Project Brief vs Business Case vs PID
Project Brief = short justification created in SU, input for IP
Business Case = living justification document updated throughout the project
PID = complete initiation documentation, created in IP, approved by the Board
💡 Trick: SU creates the Brief → IP creates the PID (which contains the full BC).
Project Assurance vs Project Support
Project Assurance = independent monitoring for the Project Board (cannot be PM!)
Project Support = administrative support for the PM (can be combined with PM)
💡 Trick: Assurance = independent control; Support = administrative help.
Risk Owner vs Risk Actionee
Risk Owner = responsible for managing, monitoring, and controlling the risk
Risk Actionee = carries out specific actions assigned by the Risk Owner
💡 Trick: Owner = risk manager; Actionee = executor.
Lessons Log vs Lessons Report
Lessons Log = continuous journal of lessons maintained throughout the project
Lessons Report = formal summary created at the end of a stage or project
💡 Trick: Log = open folder (continuous); Report = closed document (one-off).
Exception Report vs Exception Plan
Exception Report = PM informs the Board of forecasted tolerance breach
Exception Plan = PM creates a new plan upon Board request, replaces Stage Plan
💡 Trick: First Report (diagnosis) → Board requests a plan → Plan (treatment).
Management Stage vs Technical Stage
Management Stage = project segment with a Board checkpoint, plans, and tolerances
Technical Stage = grouping work by specialty (e.g., design, testing)
💡 Trick: PRINCE2 uses management stages for control (the ones with the Board at the end).

📅 2-Week Preparation Strategy

WEEK 1 — Building Knowledge (30–40 mins daily)
Day 1–2: 7 Principles — read, rewrite in own words, learn mnemonics
Day 3: 7 Themes — the questions each theme answers, key documents
Day 4–5: 7 Processes — who does what, inputs and outputs, sequence
Day 6: Roles and responsibilities — Board (Executive/SU/SS), PM, Team Manager, Assurance vs Support
Day 7: Review + 30 questions (study mode, all categories)
WEEK 2 — Practice and Consolidation (45–60 mins daily)
Day 8–9: Mock Exam (60 Qs, 60 mins) → review mistakes, analyze weak areas
Day 10: Return to theory for weak areas + practice specific categories
Day 11: Review confused terms + memorize top 10 traps (above)
Day 12: Second Mock Exam → goal: above 70% to have a buffer
Day 13: Light review — mnemonics, key definitions, 7+7+7 lists
Day 14: 🎯 EXAM — remember: 60 questions · 60 minutes · 55% pass mark
💡 Golden Rules During the Exam:
• Read the last sentence of each question first — that's the core
• When you see "accountable" → look for Project Board / Executive
• When a project loses justification → the answer is always "premature closure"
• When in doubt, pick the "most PRINCE2" answer, not "most logical"
• No negative marks — always pick an answer! Never leave blanks.
• Average 1 min per question — if stuck, mark and return later
