1. Scrum Theory — Empiricism

Scrum = framework, not methodology. Built on empiricism (learn from experience) + lean thinking (reduce waste).

3 Pillars (TIA):
• Transparency — work visible to everyone
• Inspection — artifacts & progress checked frequently
• Adaptation — adjust immediately when deviation found

5 Values (C-FORC): Commitment · Focus · Openness · Respect · Courage

Key exam facts:
• Scrum is a framework, not a process/methodology
• Definition of Ready — NOT in the Scrum Guide
• Cone of Uncertainty: early = high uncertainty, each Sprint narrows it
• 3 feedback loops: Daily Scrum, Sprint Review, Sprint Retrospective

Real life: You're building a banking app. You don't know upfront what users really need — you release a payment feature after Sprint 1, inspect real usage data (Inspection), and adapt the backlog: drop "virtual wallet" since nobody uses it (Adaptation). Empiricism in action.

2. Scrum Team — Roles & Accountabilities

One PO + one SM + Developers — max 10 people, self-managing, cross-functional.

Product Owner:
• ONE person (not a committee) — accountable for maximizing product value
• Orders Product Backlog (value, risk, learning, dependencies)
• Authority = unlimited — no CFO/CEO/architect overrides PO ordering
• PO is mandatory at Sprint Retrospective

Scrum Master:
• Accountable for Scrum effectiveness & removing impediments
• Serves PO, Developers, and the organization

Developers:
• Create Done Increments & own the Sprint Backlog
• Self-organize — no one tells them how to work
• Estimate PBIs (PO does NOT estimate)

Real life: The CTO tells the PO to put "refactor authentication" at the top of the backlog. The PO says no — a customer-facing payment bug is far more valuable to fix first. The CTO must respect this: PO authority is unlimited. The SM helps the PO communicate this decision to stakeholders.

3. Scrum Events

All events = inspect & adapt opportunities. Timeboxed = max duration, can end earlier.

EventTimebox (1-mo Sprint)Who
Sprint≤ 1 monthScrum Team
Sprint Planning≤ 8hScrum Team
Daily Scrum15 minDevelopers
Sprint Review≤ 4hScrum Team + stakeholders
Sprint Retrospective≤ 3hScrum Team (PO mandatory!)
Exam traps:
• Review BEFORE Retrospective — always
• Sprint Review ≠ demo/sign-off — it's inspect & adapt the product
• Retro ≠ Developers only — full Scrum Team (PO required)
• Daily Scrum = Developers only (SM optional, PO not invited)
• Stakeholders attend Sprint Review only — not Planning/Daily/Retro
• No pre-conditions for Sprint Planning defined in Scrum Guide
• Sprint cancelled ONLY if Sprint Goal becomes obsolete (PO decides)
• New Sprint starts IMMEDIATELY after previous — zero gap

Real life: Sprint ends Friday. Sprint Review happens — the team shows a live checkout flow to the sales VP (stakeholder). Monday morning: Sprint Retrospective — the team (PO included) decides to write more automated tests after a QA incident. New Sprint Planning starts Monday afternoon. Zero gap.

4. Scrum Artifacts & Commitments

ArtifactCommitment
Product BacklogProduct Goal
Sprint BacklogSprint Goal
IncrementDefinition of Done
Key facts:
• Product Backlog is ordered (not "prioritized") — evolves forever, never complete
• Sprint Goal = WHY of the Sprint — fixed for Sprint duration
• Sprint Backlog = Goal + selected PBIs + plan — can adapt daily
• Increment must meet DoD — PO decides IF/WHEN to release it
• Multiple Increments can be created in one Sprint
• DoD: updated at Sprint Retro — can only get stricter, never looser
• DoD ≠ Acceptance Criteria — DoD = quality floor for ALL; AC = specific to one PBI
• Undone work → returns to Product Backlog

Real life: Developers finish a "dark mode" feature but skip security review (part of DoD). The PO cannot release it — DoD is not met. It goes back to the backlog. At the Retrospective, the team adds "penetration test" to the DoD to prevent recurrence.

5. Product Vision & Strategy

Vision = long-term purpose (not in Scrum Guide but critical for PO).
Product Goal = medium-term target IN the Product Backlog — must finish or abandon before taking next one.
Roadmap = communication tool showing product evolution — not prescribed by Scrum, owned by PO.

PO balances 3 horizons:
• Long-term: Vision
• Medium-term: Product Goal
• Short-term: Sprint Goal

Exam tip: Product Goal is IN the Scrum Guide (2020). It is the commitment of the Product Backlog. One goal at a time.

Real life: Vision = "be the #1 B2B invoicing platform in Europe." Product Goal = "reduce invoice processing time to under 2 minutes." Current Sprint Goal = "allow bulk invoice upload." Every PBI in this Sprint must connect to that chain — or it shouldn't be in the Sprint at all.

6. Product Value

PO accountable for maximizing value — Scrum Guide doesn't prescribe how to measure it.

Output vs Outcome:
• Output = features, velocity, story points (internal metrics)
• Outcome = revenue, user adoption, NPS, cost reduction (value)
Always choose outcomes over outputs on exam.

Backlog ordering factors: Value · Risk · Dependencies · Learning · Cost of delay

TCO (Total Cost of Ownership): PO considers full lifecycle — build + run + maintain. Cheap to build ≠ good investment if expensive to support.

Value realized only when Increment reaches users — not when Sprint ends.

Real life: Team ships a notification feature. Velocity went up. But users disabled notifications immediately — adoption is 3%. PO measures outcome (adoption, NPS) not output (velocity). Next Sprint: investigate why and build an opt-in preferences screen instead of more features.

7. Product Backlog Management

Refinement = ongoing — no timebox in Scrum Guide (typically ≤10% of Sprint).
NOT a formal Scrum event. Developers collaborate with PO during refinement.

Key facts:
• Ordered not "prioritized" — ordering = multi-factor decision
• Definition of Ready — NOT in Scrum Guide, not required
• Estimates made by Developers (not PO, not SM)
• PO can add new urgent items any time — then orders appropriately
• Top items = more refined, smaller, better understood
• Technical debt = in the Product Backlog, PO orders it like any other item

Real life: A compliance team reports a new EU regulation kicks in 3 weeks. PO adds "GDPR consent flow" to the Product Backlog and moves it near the top. Developers refine it with the PO — they break it into 3 PBIs and estimate during a refinement session. No "Definition of Ready" gate needed — they just make sure it's understood.

8. Stakeholder Management

PO = interface between team and stakeholders. Represents stakeholder interests but makes independent ordering decisions.

Who attends what:
• Sprint Review — YES (invited by PO)
• Daily Scrum / Sprint Planning / Retrospective — NO
• Stakeholders can talk to anyone anytime — just not in those events

PO protects team from: ad-hoc requests that bypass the Product Backlog.
To change the backlog: stakeholders must convince the PO — they cannot change it directly.

Real life: The Head of Sales corners a developer on Slack: "Can you add a CSV export this week? I need it for a demo." Developer replies: "Talk to the PO — everything goes through the Product Backlog." PO evaluates, adds it, and orders it for next Sprint. Team is protected from direct stakeholder pressure.

9. Release Management

Release anytime — PO decides if/when to release a Done Increment. Not tied to Sprint boundaries.

Key facts:
• "Potentially releasable" = meets DoD — user-ready
• Undone work cannot be released, cannot be shown at Sprint Review
• CI/CD = not prescribed by Scrum, but strongly complementary
• Technical debt: limits future value delivery — PO must understand it, manage it via backlog
• Multiple Done Increments can exist before release

Real life: Sprint 4 ends. Two features are Done (meet DoD). PO decides to release one immediately (a bug fix users are waiting for) and hold the other (a new dashboard) for a coordinated marketing launch next month. Both are "potentially releasable" — PO just chose the right timing for business reasons.

10. Scrum in Practice

Multiple teams, same product:
• One Product Backlog · One Product Owner — always
• Same Sprint cadence (same start date) for alignment
• Scaling framework: Nexus (3–9 teams, Scrum.org)

Mid-Sprint problems:
• Scope too large → PO + Devs renegotiate Sprint Backlog scope; Sprint Goal stays
• Sprint Goal becomes obsolete → only PO can cancel Sprint
• Sprint 0 = does NOT exist in Scrum

Continuous improvement:
• Retro = formal improvement event; DoD becomes stricter over time
• At least 1 improvement from Retro → added to next Sprint Backlog

Real life: 3 teams build the same e-commerce platform. All start Sprint 5 on Monday. One PO orders a single shared backlog. Teams pick their PBIs at Sprint Planning. At Sprint Review on Friday, all three present to the same stakeholders. Integration issues are visible and fixed before Sprint 6.

11. Evidence-Based Management (EBM)

Complementary Scrum.org framework — PO uses it to measure value and make decisions.

4 Key Value Areas (KVAs):
• CV (Current Value) — value delivered today: revenue, NPS, satisfaction
• UV (Unrealized Value) — gap between current & desired state
• A2I (Ability to Innovate) — reduced by tech debt, legacy systems
• T2M (Time-to-Market) — how fast new value reaches users

Exam tip: EBM = outcomes not outputs. Velocity/story points measure output. NPS/revenue measure outcome. PO success = value outcomes, not team speed.

Real life: PO's boss says "velocity went from 30 to 45 — great job!" PO replies: "Revenue is flat and churn increased. We shipped more but built the wrong things. Let's look at UV — 60% of users still can't complete checkout on mobile. That's our next target."

12. Exam Traps — Quick Kills

Remember these cold:
• Backlog = ordered (never "prioritized")
• PO = 1 person; unlimited backlog authority; no exceptions
• Only PO cancels a Sprint (when goal is obsolete)
• Developers estimate — PO does not
• DoD created by Scrum Team (if no org standard) — updated at Retro only
• Sprint Goal = fixed; Sprint Backlog = flexible
• Sprint Review = working session, NOT demo/approval/sign-off
• Refinement = no timebox, NOT a Scrum event
• Scrum does NOT prescribe: user stories, story points, velocity, burn-down, DoR
• Multiple teams = one PO + one Product Backlog + same Sprint cadence
• PO required at Sprint Retro (mandatory, not optional)
• New Sprint = IMMEDIATELY after previous, zero gap
• Value ≠ revenue only (NPS, adoption, cost reduction, strategic position)
• Sprint 0 = does not exist in Scrum
• TCO = PO owns full lifecycle: build + run + maintain
• Stakeholders: Sprint Review only (not Planning/Daily/Retro)
• Definition of Ready = NOT in Scrum Guide
• Velocity increase ≠ PO success (outcome matters, not output)
